Logstash管道介绍

Logstash 是免费且开放的服务器端数据处理管道，拥有200多个插件，能够从多个来源采集数据，转换数据，然后将数据发送到喜欢的“存储库”中。在本文中主要实践的是将数十亿的数据从MySQL数据库如何高效导入ES

logstash管道必须要有input和output元素，filter元素是可选的。input插件定义数据来源，filter插件用来修改用户指定的数据，output插件定义数据写入何地。

logstash结构如下所示：

input：采集各种样式、大小和来源的数据
Logstash 支持各种输入选择 ，同时从众多常用来源捕捉事件。
能够以连续的流式传输方式，轻松地从日志、指标、Web 应用、数据存储以及各种服务中采集数据。

filter：实时解析和转换数据；数据从源传输到存储库的过程中，Logstash 过滤器能够解析各个事件，识别已命名的字段以构建结构，并将它们转换成通用格式，以便更轻松、更快速地分析和实现商业价值。
利用 Grok 从非结构化数据中派生出结构
从 IP 地址破译出地理坐标
将 PII 数据匿名化，完全排除敏感字段
简化整体处理，不受数据源、格式或架构的影响

output：选择目标存储库，可以选择ElasticSearch也可以选择其他存储库

Logstash管道文件配置

logstash是通过jdbc input插件来读取mysql中的数据，配置文件如下：

参数说明

input配置

jdbc_connection_string*：数据库类型、地址、以及数据库名称

jdbc_user： 数据库用户名

jdbc_password： 数据库密码

jdbc_driver_library：jdbc驱动文件地址： https://static.denglin.me/blog/files/com.mysql.jdbc_5.1.5.jar

statement：查询数据的sql语句；

jdbc_paging_enabled：是否启用分页，如果您的数据较多，建议启用分页，不然logstash会按照您填写的sql语句一次加载所有数据到内存中，有可能造成内存不足；

jdbc_page_size： 启用分页后每次读取数据量

output配置

output插件这里是elasticsearch

hosts: 是es的访问地址，这里可以填写多个节点的地址

index: 数据要保存到的索引名称

document_type：数据要保存到的type名

workers：批量索引的线程数，和CPU数量一致

优化传输

性能分析

logstash提供了丰富的调测命令:

  1. event可以查看当前处理的in， filter，output的累计处理数量，如果in 数量大于filter和out，说明filter和out有性能瓶颈，如果filter大于out说明out有性能瓶颈。

  1. jvm可以查询堆内存使用情况

  2. pipeline查询处理的详细信息

  3. 通过hot_threads判断logstash热点线程位置。

也可以通过一下方式做性能分析

  1. cat /proc/cpuinfo |grep "processor"|wc -l 查看CPU核数
  2. cat /proc/cpuinfo 查看CPU信息
  3. top命令查看读取与写入的PID和CPU占用情况 ，top 命令用于实时显示 process 的动态。
  4. ps aux | grep PID 确认PID是Logstash
  5. top -Hp PID 查看线程运行情况
  6. 查看线程对CPU的使用情况

性能优化

logstash也有一些调整参数，

方法一：增加线程

logstash work线程数量，通常等于vcpu的数量或者整数倍， 如果cpu利用率低可以增加work数量，增大并发数据。

./bin/logstash -f config/logstash-test.conf -w 128 & # 默认为64，开启128个线程

方法二：设置一批次写入ES的doc数量

一批次写入ES的doc数量，设置太小会频繁的消耗ES写入IO，对于大流量写入数据配置大一些

./bin/logstash -f config/logstash-test.conf -b 300 & # 默认batchsize为125，可以增加为300

方法三：设置delay时间

超时时间，如果设定时间没有收到新的event，logstash开始执行filter和output，logstash接收event到event被处理等待的最大时间是pipeline.batch.size * pipeline.batch.delay。

./bin/logstash -f config/logstash-test.conf -u 10 & # 默认delay时间为50ms，可以减

文件配置

也可以将以上配置建议直接在logstash.yaml 中配置

logstash调优主要是减少event等待时间加快线程pipeline的切换。

方法四：解决MySQL的分页偏移导致的低效

如图： 数据量超过一亿，偏移量特别大的时候的查询时间，会超过4分钟，不可忍受！

导致缓慢的原因时当offset时一亿，limit是二十万时，MySQL的查询需要一次查询出offset+limit的数据，然后将offset之前全部丢弃，然后取得一亿至一亿二十万的数据，导致offset越大MySQL响应越慢。

解决方案：

  1. 设置条件查询，也就是说一亿两千万条的数据，可以设置where ID < 120000000 AND　ID >= 100000000 然后对此条件范围内的两千万数据分页查询。

方法五：提升CPU的使用

  1. 线程增加之后发现CPU的使用率还是很小，无压力

  2. 更具命令查看in，filter，out三部分的处理数量，可以看到filter 和out的处理量是几乎等于in的数量，也就是说filter和out不存在瓶颈，因此效率瓶颈在in，也就是查询效率比较低效

  3. 设置多个INPUT同步查询，提高查询效率，提高CPU使用率

​ INPUT１：where ID < 10000000

​ INPUT2 : where ID < 20000000 AND ID >= 10000000

​ …

​ INPUT5: …

​ 注意：多个INPUT的LIMIT总和不高于内存限制
