ElasticSearch概述

Elasticsearch（ES）是一个基于Lucene （Lucene是apache下的一个开源的全文检索引擎工具包。提供一个简单易用的工具包（类库），方便的在目标系统中实现全文检索的功能。）构建的开源、分布式、RESTful接口的全文搜索引擎。Elasticsearch还是一个分布式文档数据库，其中每个字段均可被索引，而且每个字段的数据均可被搜索，ES能够横向扩展至数以百计的服务器存储以及处理PB级的数据。可以在极短的时间内存储、搜索和分析大量的数据

ES基本概念

Cluster集群

ES可以作为一个独立的单个搜索服务器。同时，为了处理大型数据集，实现容错和高可用性，ES可以运行在许多互相合作的服务器上，这些服务器的集合称为集群。

Node节点

形成集群的每个服务器称为节点。

Index 索引

ES将数据存到索引之中，索引存放和读取的单元是Document文档。与关系型数据库中的数据库实列database是类似的。

注：ElasticSearch 6.x 版本之前还有Type ，6.x 版本后就将其废弃，建议的是每个类型（业务）的数据单独放在一个索引中，这样其实回归到一般意义上的索引定义，索引定位文档。

Document文档

在ES中Document文档是最基础的可被索引的数据单元，类似于一个JSON格式数据

Shards分片

number_of_shards值指定索引的分片数量，而不与整个集群有关。在一个搜索里存储的数据，潜在的情况下可能会超过单个节点的硬件的存储限制，因此，elasticsearch提供了分片的功能，它可以将索引划分为多个分片，当你创建一个索引的时候，你就可以简单的定义你想要的分片的数量，每一个分片本身是一个全功能的完全独立的索引，可以部署到集群中的任何一个节点。Elasticsearch自动管理这些分片的排列。它还会根据需要重新平衡分片，无需担心细节。

分片好处：

（1）允许水平切分索引，然后分布在多台机器上，提升效率，降低硬件存储限制

（2）分片可以通过分布和并行操作来应对日益增长的执行量

Replicas副本

副本是提供了一个故障恢复机制，ES允许制作一个或多个拷贝放入一个叫做复制分片中。他有两个重要作用

（1）高可用。它提供了高可用的以来防止分片或者节点宕机，为此，一个非常重要的注意点就是绝对不要讲一个分片的拷贝放在跟这个分片相同的机器上。

（2）高并发。它允许你的分片可以提供超出自身吞吐量的搜索服务，搜索行为可以在分片所有的拷贝中并行执行。

#ES存储过程

写入过程

jjjjjjj

  1. 客户端选择一个node发送请求过去，这个node就是coordinating node (协调节点)

  2. coordinating node，对document进行路由，将请求转发给对应的node

  3. 实际上的node上的primary shard处理请求，将一份数据写入primary shard,它会将分成成对的shard分片，并将数据进行复制

  4. coordinating node，如果发现primary node和所有的replica node都搞定之后，就会返回请求到客户端

    更多底层写入过程：https://segmentfault.com/a/1190000015256970

查询过程

  1. 客户端发送一个请求给coordinate node

  2. 协调节点将搜索的请求转发给所有的shard对应的primary shard 或replica shard

  3. query phase：每一个shard 将自己搜索的结果（其实也就是一些唯一标识），返回给协调节点，有协调节点进行数据的合并，排序，分页等操作，产出最后的结果

  4. fetch phase ，接着由协调节点，根据唯一标识去各个节点进行拉去数据，汇总返回给客户端

    更多底层查询过程：https://segmentfault.com/a/1190000015256970

故障恢复过程

ES客户端取数据的时候就会在replica或primary 的shard中去读。ES集群有多个节点，会自动选举一个节点为master节点，这个master节点其实就是干一些管理类的操作，比如维护元数据，负责切换primary shard 和replica shard的身份之类的，要是master节点宕机了，那么就会重新选举下一个节点为master为节点。如果时非master宕机了，那么就会有master节点，让那个宕机的节点上的primary shard的身份转移到replica shard上，如果修复了宕机的那台机器，重启之后，master节点就会控制将缺失的replica shard 分配过去，同步后续的修改工作，让集群恢复正常。

查询原理

倒排序索引

在ES搜索引擎中，每个文档都有一个对应的文档 ID，文档内容被表示为一系列关键词的集合。例如，文档 1 经过分词，提取了 20 个关键词，每个关键词都会记录它在文档中出现的次数和出现位置。也就是说，倒排索引就是关键词到文档 ID 的映射，每个关键词都对应着一系列的文件，这些文件中都出现了关键词。比如：有以下5个文档

倒排索引还可以记录更多的信息，比如文档频率信息，表示在文档集合中有多少个包含某个单词。

根据倒排索引，搜索引擎可以很方便地响应用户的查询。比如用户输入查询 Facebook，搜索系统查找倒排索引，从中读出包含这个单词的文档，这些文档就是提供给用户的搜索结果。

ES特点和优势

1）分布式实时文件存储，可将每一个字段存入索引，使其可以被检索到。

2）实时分析的分布式搜索引擎。

分布式：索引分拆成多个分片，每个分片可有零个或多个副本。集群中的每个数据节点都可承载一个或多个分片，并且协调和处理各种操作； 负载再平衡和路由在大多数情况下自动完成。

3）可以扩展到上百台服务器，处理PB级别的结构化或非结构化数据。也可以运行在单台PC上（已测试）

4）支持插件机制，分词插件、同步插件、Hadoop插件、可视化插件等。
